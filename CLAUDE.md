# CLAUDE.md — Aegis Mail Developer Guide

Complete reference for AI coding tools continuing development on this codebase.

---

## Project Overview

**Aegis Mail** is a Chrome Extension (Manifest V3) that injects into Gmail and provides:

- Right-side sliding panel: categorizes inbox emails with unread/all toggle
- Security analysis popup: auto-triggers when opening an email (safety score + issues + link audit)
- Dual analysis modes: local keyword rules (default, no API needed) or OpenAI-compatible AI
- Bulk email operations: checkbox select → delete / move-to-label via simulated DOM events
- Domain whitelist validation: detects off-brand links and spoofing attempts
- No Gmail OAuth required — all operations through DOM event simulation

**Tech stack:** Vanilla JavaScript ES6+, Chrome MV3, Jest (unit tests), Makefile (build)

### Content Script Load Order (manifest.json — DO NOT change order)

```
whitelist-manager.js → email-analyzer.js → ai-analyzer.js
→ base-platform.js → gmail-platform.js
→ analysis-panel.js → email-popup.js → content.js
```

Each module exports itself to `window` (e.g. `window.EmailAnalyzer`). `content.js` uses global variables directly.

## Message Passing Architecture

```
popup.js  ──PING/ANALYZE──▶  content.js  ──GET_SETTINGS──▶  background.js
                                         ──AI_ANALYZE────▶
                                         ──GET_WHITELIST──▶
options.js ──SAVE_SETTINGS──────────────────────────────▶
```

| Message Type | Direction | Purpose |
|---|---|---|
| `PING` | popup → content | Check if content script is alive |
| `ANALYZE` | popup → content | Trigger email list analysis + show panel |
| `GET_SETTINGS` | content → background | Fetch chrome.storage.sync settings |
| `AI_ANALYZE` | content → background | Single email AI categorization (CORS proxy) |
| `AI_BATCH_ANALYZE` | content → background | Batch email AI categorization |
| `GET_WHITELIST` | content → background | Fetch cached whitelist from storage.local |
| `FETCH_WHITELIST` | content → background | Force-fetch whitelist from remote URL |
| `GET_WHITELIST_STATUS` | content → background | Get whitelist metadata (count, last updated) |
| `RESOLVE_SHORT_URL` | content → background | Follow short URL redirect, return final URL |
| `SAVE_SETTINGS` | options → background | Write settings to chrome.storage.sync |
| `TEST_AI_API` | options → background | Test AI API connectivity |
| `FETCH_AI_MODELS` | options → background | List available models from AI endpoint |

## Settings Schema (chrome.storage.sync)

Defined in `background.js` → `DEFAULT_SETTINGS`:

```javascript
{
  analysisMode: 'local' | 'ai',
  analysisDebug: false,           // enable debug logging (window.__aegisDebug)
  aiSettings: {
    baseUrl: 'https://api.openai.com/v1',
    apiKey:  '',
    model:   'gpt-5-nano-2025-08-07',  // any OpenAI-compatible model ID
  },
  categories: [
    {
      id:       string,      // unique key, used as Map index
      name:     string,      // display name (supports CJK)
      emoji:    string,      // emoji name key (e.g. 'briefcase') or literal emoji
      color:    string,      // hex text/border color
      bgColor:  string,      // hex background color
      keywords: string[],    // case-insensitive match strings
      isCustom?: boolean,    // true for user-created categories
      createdAt?: number,    // timestamp for user-created categories
    }
  ]
}
```

**7 default categories:** work (工作), shopping (購物), finance (財務), social (社交), promotions (促銷), security (安全), notifications (通知)

**Storage limits:** 8KB per item, 100KB total for `chrome.storage.sync`. If keyword count grows large, consider migrating to `chrome.storage.local` (5MB).

The whitelist is stored separately in `chrome.storage.local`:
- `whitelistData`: full whitelist JSON object
- `whitelistLastUpdated`: timestamp
- `whitelistUrl` (in sync): optional remote URL for auto-refresh (weekly alarm `aegis-whitelist-update`, `periodInMinutes: 10080`)

## Safety Score Logic (`email-analyzer.js: analyzeEmailDetail`)

Starts at 100. Deductions:

| Check | Deduction | Cap |
|---|---|---|
| No sender email address | -20 | — |
| Temp/disposable email service (from whitelist `suspiciousDomains`) | -30 | — |
| Local part has 4+ consecutive digits | -10 | — |
| Suspicious TLD (.xyz .top .click .loan .work .date .win .bid .stream) | -15 | — |
| Auto-generated local part (12+ alphanumeric chars, no symbols) | -5 | — |
| Phishing keywords in subject+body | -10/each | -30 |
| HTTP links (not HTTPS) | -5/each | -15 |
| Potential spoofing (keyword match but sender domain doesn't match service) | -25/each | -40 |
| Off-whitelist links (sender matches service but link domain doesn't) | -15/each | -30 |
| Suspicious domain links (IP address, suspicious TLD) | -15/each | -30 |

| Score | Level | Color |
|---|---|---|
| 80–100 | safe | `#1a7f37` |
| 50–79 | caution | `#9a6700` |
| 0–49 | danger | `#cf222e` |

## Whitelist System (`src/data/service-whitelist.json`)

Schema:
```json
{
  "version": "2",
  "updatedAt": "...",
  "services": [
    {
      "id": "google",
      "name": "Google",
      "keywords": ["google", "gmail"],
      "baseDomains": ["google.com", "gmail.com"],
      "senderDomains": ["google.com", "gmail.com"]
    }
  ],
  "publicEmailDomains": ["gmail.com", "yahoo.com", ...],
  "suspiciousDomains": ["tempmail.com", "guerrillamail.com", ...],
  "shortUrlServices": ["bit.ly", "tinyurl.com", ...]
}
```

**WhitelistManager methods:**
- `init()` — load from background cache (call on content script start)
- `findServiceBySenderDomain(senderEmail)` — match sender's domain to a service
- `findServicesByKeywords(emailContent)` — keyword-based service detection (for spoofing)
- `isDomainInService(hostname, service)` — check if link belongs to a service's baseDomains
- `isKnownShortUrl(url)` — check if URL uses a known shortener
- `isPublicEmailDomain(domain)` — detect personal email services
- `isSuspiciousDomain(domain)` — detect temp/disposable email services
- `extractBaseDomain(hostname)` — handles .co.jp, .com.tw special TLDs

**Domain matching rules:**
- Base domain matching: `mail.google.com` → `google.com`
- Special TLD handling: `service.ntpc.gov.tw` matches `gov.tw`
- Subdomain matching: `subdomain.example.com` matches `example.com`

## Startup Flow

**First install:**
```
chrome.runtime.onInstalled
  → background.js reads chrome.storage.sync
  → writes DEFAULT_SETTINGS for any missing keys
  → seeds bundled whitelist to chrome.storage.local
  → creates weekly alarm 'aegis-whitelist-update'
```

**User opens Gmail (new tab):**
```
Chrome auto-injects content scripts (manifest order)
  → content.js IIFE runs
  → checks window.__aegisInitialized (prevents duplicate injection)
  → new GmailPlatform()
  → platform.isMatchingPage() confirms Gmail
  → WhitelistManager.init() loads whitelist
  → platform.observeNavigate() watches SPA route changes
  → if email detail URL → analyzeOpenEmail() after 1.2s delay
```

**User clicks extension icon on existing Gmail tab:**
```
popup.js
  → sendMessage PING → if no response: dynamically inject all scripts + CSS
  → sendMessage ANALYZE → content.js: runAnalysis()
```

## AI Mode

**Single email analysis (`AI_ANALYZE`):**
- Background fetches `{aiSettings.baseUrl}/chat/completions`
- System prompt: classify into one category from the provided list
- User message: Subject + From + Body[:1000] + Links[:10]
- Returns `{ category: "category name" }`

**Batch analysis (`AI_BATCH_ANALYZE`):**
- Sends array of `{ id, subject, sender, senderEmail }`
- Returns `{ results: [{ id, category }] }`

**Model list fetching (`FETCH_AI_MODELS`):**
- OpenAI-compatible: `GET /models` with Authorization header
- Gemini: `GET /v1beta/models?key=API_KEY`

## Development Workflow

### Setup
```bash
npm install          # install Jest and test dependencies
make dev             # copy source → dist/ (excludes .git, node_modules, __tests__, etc.)
```

### Load in Chrome
1. `make dev`
2. Go to `chrome://extensions` → Enable Developer Mode
3. "Load unpacked" → select `dist/`
4. After code changes: `make dev` → click reload icon on extensions page

### Build Commands
```bash
make dev    # development build to dist/
make zip    # dist/aegis-mail-{version}.zip for Chrome Web Store
make crx    # dist/aegis-mail-{version}.crx for self-distribution
make clean  # remove dist/
make info   # show version, Chrome path
```

Version is read from `manifest.json` → `version` field.

### Testing
```bash
npm test                    # run all 65 tests
npm run test:coverage       # with coverage report → coverage/lcov-report/
npm run test:watch          # watch mode
npm run test:analyzer       # test only email-analyzer
npm run test:whitelist      # test only whitelist-manager
npm run test:ai             # test only ai-analyzer
```

Tests run in jsdom with Chrome API mocks (`__tests__/setup.js`). No real browser or API calls needed.

### Debug Mode
Set `analysisDebug: true` in settings (options page or DevTools):
```javascript
chrome.storage.sync.set({ analysisDebug: true })
```
Then open Gmail DevTools Console — verbose grouped output labeled `[Aegis Debug]` will appear during analysis.

## Extending the Extension

### Add a New Email Category

In `background.js` → `DEFAULT_SETTINGS.categories[]`:
```javascript
{
  id: 'travel',           // unique, lowercase with hyphens
  name: '旅遊',
  emoji: '✈️',
  color: '#0288d1',
  bgColor: '#e1f5fe',
  keywords: ['flight', '機票', 'hotel', '飯店', 'booking', '訂房'],
}
```

Users can also add/edit/remove categories via the options page.

### Add a New Platform (Yahoo Mail, Outlook)

1. Create `src/platforms/yahoo-platform.js` extending `BasePlatform`
2. Implement all abstract methods: `getName()`, `isMatchingPage()`, `getEmails()`, `getLabels()`, `getEmailDetail()`, `deleteEmails()`, `moveToLabel()`, `observeNavigate()`
3. Add URL pattern to `manifest.json` → `content_scripts[0].matches`
4. Add `src/platforms/yahoo-platform.js` to `manifest.json` content_scripts load order (before `content.js`)
5. In `content.js`, select platform by URL:
```javascript
const platform = window.location.href.includes('yahoo.com')
  ? new YahooPlatform()
  : new GmailPlatform();
```

### Add Phishing Keywords

Edit `PHISHING_KEYWORDS` array at top of `src/analysis/email-analyzer.js`.

### Add Whitelist Services

Add entries to `src/data/service-whitelist.json` → `services[]`. Each service needs `id`, `name`, `keywords[]`, `baseDomains[]`, `senderDomains[]`.

### Add Gmail Locale Support

In `gmail-platform.js` → `GMAIL_ACTIONS`, add the locale's tooltip strings to `tooltips[]` and `ariaLabels[]`. For substring matching, add to `tooltipContains[]`.

## Known Limitations

### Gmail DOM Fragility
Gmail is a complex SPA; DOM operations are unsupported and may fail:
- **Checkbox not responding**: `_selectRows()` dispatches `mouseenter`/`mouseover` to force checkbox render, but Gmail's virtual DOM may still not render it
- **Toolbar delay**: `_waitForElement()` polls for 3s max, then falls back to keyboard shortcut `#` (Shift+3) for delete
- **Class name changes**: update `GMAIL_ACTIONS` and `getEmails()` selectors if Gmail updates break functionality

### Content Script Injection Timing
If Gmail was open before the extension was installed, Chrome won't auto-inject. `popup.js` → `ensureContentScript()` handles this via dynamic injection fallback.

### CORS for AI API
AI API calls must go through `background.js` (service worker). Direct fetch from content script will be blocked by CORS.

### Storage Quota
`chrome.storage.sync`: 8KB per item, 100KB total. Heavy keyword usage may need migration to `chrome.storage.local` (5MB limit).

### Gmail SPA Navigation
URL hash changes are monitored via `MutationObserver` on `document.title` + `hashchange` events. Email detail pages are detected by regex `/^#[^/]+\/[A-Za-z0-9]{10,}/`.

## Code Conventions

- **Classes**: PascalCase (`AnalysisPanel`, `GmailPlatform`)
- **Functions**: camelCase (`analyzeEmailList`, `getSettings`)
- **Constants**: UPPER_SNAKE_CASE (`PHISHING_KEYWORDS`, `SUSPICIOUS_TLDS`)
- **CSS**: kebab-case with `aegis-` prefix (`.aegis-panel`, `#aegis-email-popup`)
- **All modules**: IIFE pattern with `window.ModuleName` export
- **DOM selectors**: prefer `[role="..."]` attributes over class names for stability
- **Error handling**: try-catch all async ops; fall back to local analysis if AI fails; `console.warn` for non-critical errors; alert only for actionable failures
